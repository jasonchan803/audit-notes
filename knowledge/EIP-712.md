## EIP-712 结构化签名

**一句话定义**：让用户在钱包里签一个“看得懂”的结构化数据（比如“你要授权 Uniswap 用 100 USDC”），而不是一串乱码。

**核心目的**：防止签名被重放到错误的合约/链/版本上。

**EIP-712 签名的消息体为**：\x19\x01（EIP-712 固定前缀）+ domain separator（域分隔符）+ structHash（结构哈希），然后对该消息体进行哈希并签名。

**domain separator（域隔离）是什么**：这是每个 DApp 唯一的“签名上下文”，用来**锁定签名的使用场景**，保证它不会在错误的链、错误的合约、错误的版本里被意外接受，domain separator 是由 domainSeparatorHash = keccak256(abi.encode(domainTypehash, name, version, chainId, verifyingContract)) 计算得出的。

**domain separator 里包含哪些关键信息**：
- `domainTypehash` ：类型哈希（确认下列参数的类型），等于 keccak256("EIP712Domain(string name,string version,uint256 chainId,address verifyingContract)")
- `name`：DApp 名称
- `version`：版本号
- `chainId`：链 ID（防止跨链重放）
- `verifyingContract`：验证签名的合约地址

**为什么 `version` 要随着新版本合约递增**：旧合约的签名永远无法在新合约上通过验证，彻底杜绝版本混淆。

**structHash（结构哈希）是什么**：在普通消息的基础上结构化，固定每个消息的类型，同时保留值的动态传入，structHash = keccak256(abi.encode(typehash, ...values))，其中 typehash 是自定义结构体的类型哈希（如 keccak256("Person(string name,address wallet)")）

**structHash 里包含哪些关键信息**：
- `Typehash` ：类型哈希（确认下列参数的类型），同样是一个常量，与上述的`domainTypehash`类似
- `Type`：根据自身业务逻辑设计固定的类型
- `Value` ：动态数据（传入到设计好的固定类型当中）

**常见 EIP-712 相关漏洞**：
- Domain separator 缺字段（缺少 chainId、version 等）→ 签名可能在别的链或别的版本上被复用
- Typehash 写错 → 合约无法正确还原签名，接入方签出来的东西跟合约不匹配

**Code (openzeppelin提供返回数据到前端的函数)**:

```solidity
    /// @inheritdoc IERC5267
    function eip712Domain()
        public
        view
        virtual
        returns (
            /*
            fields标记读取以下的那些参数，如fields返回 0x0f（二进制 01111），
            那么它只会处理前4个核心字段，name,version,chainId,verifyingContract,不包含后面的salt和extensions
            因为签名在前端，所以一定要确保前端读取的参数和后端传入计算的参数一致，否则前后端的签名不一致会导致验证失败
            90%的项目会通过前端的硬编码来固定参数而非通过后端返回，这样能确保domain separator固定，保证签名的稳定性
            当合约需要升级的时候，前端同步更新就行，且前端的修改成本远低于后端，这就是目前最佳的应用方案
            */
            bytes1 fields,
            string memory name,
            string memory version,
            uint256 chainId,
            address verifyingContract,
            bytes32 salt,
            uint256[] memory extensions
        )
    {
        return (
            /*
            openzeppelin目前的标准就是只返回前4个核心字段，不包含后面的salt和extensions
            */
            hex"0f", // 01111
            _EIP712Name(),
            _EIP712Version(),
            block.chainid,
            address(this),
            bytes32(0),
            new uint256[](0)
        );
    }
```

**但实际现在90%项目都不会调用该函数，为了稳定与安全性，大部分项目会选择在前端直接硬编码来代替使用通过调用后端返回数据到前端的方法，但openzeppelin为了接口设计了通用标准，即使 90% 的人用不到，我也要为 10% 的人准备，硬编码方式虽然常见，但要求前端与合约严格同步版本；若合约升级后 version 变化而前端忘记更新，则所有旧签名立即失效。这正是审计师需要关注的风险点。
**
